=== Engine Internals & Extension Points

This section covers how to extend and modify the FIFE engine internals.

==== Adding a New Engine Subsystem

To add a new subsystem to the engine:

1. Create a new directory under `engine/core/` (e.g., `engine/core/mysubsystem/`).
2. Define a header with a class inheriting from no base (subsystems are standalone).
3. Register the subsystem in `Engine::createSubsystems()` in `engine/core/engine.cpp`.
4. Add initialization in `Engine::init()` and cleanup in `Engine::destroy()`.
5. If the subsystem needs configuration, add entries to `EngineSettings`.
6. Create a SWIG interface file in `engine/swigwraps/` to expose it to Python.

==== Exposing New C++ Methods

To expose a new C++ method to Python:

1. Add the method declaration to the C++ header (e.g., `engine/core/model/mymethod.h`).
2. Add a `%include` directive to the corresponding `.i` file (e.g., `model.i`):
+
[source,swig]
----
%include "model/mymethod.h"
----
3. Rebuild the project. SWIG regenerates the Python wrapper automatically.

==== Memory Ownership in SWIG

SWIG-generated wrappers use reference counting. Key rules:

* Objects returned by value are copied into Python-owned memory.
* Objects returned by pointer are managed by C++ unless `%newobject` is specified.
* Use `%newobject` when Python should take ownership and delete the object:
+
[source,swig]
----
%newobject Model::createMap;
----

==== Resource Manager Internals

FIFE uses manager classes to cache and lazily load resources:

[cols="1,2,2", options="header"]
|===
| Manager | Resource Type | Caching Strategy

| `ImageManager`
| Textures and sprites
| LRU cache with configurable max size

| `AnimationManager`
| Animation sequences
| Loaded on demand, retained until explicitly released

| `SoundClipManager`
| Audio clips
| Loaded on first playback, cached until release
|===

Resource managers follow a common pattern:

1. Request a resource by ID: `imageManager->get(resourceId)`
2. Manager checks cache â€” if found, returns cached resource
3. If not found, loads from VFS, caches, and returns
4. Call `release(resourceId)` to free from cache

==== Event/Listener Pattern

FIFE uses an observer pattern for events:

[source,cpp]
----
class MyListener : public fife::MouseListener {
public:
    void mousePressed(fife::MouseEvent& evt) override {
        // Handle mouse press
    }
};

// Register listener
eventManager->addMouseListener(myListener);
----

Available listener types: `MouseListener`, `KeyListener`, `WidgetListener`, `TimeEventControllerListener`.

==== Renderer Pipeline

Renderers are registered and executed each frame in order:

1. Camera queries visible instances from the model
2. Each registered renderer processes instances (grid, generic, cell, coordinate)
3. Renderers are sorted by priority (lower values render first)
4. Each renderer calls `setColor/getImage` on instances and draws to the camera's render target

To register a custom renderer, subclass `RendererBase` and call `View::addRenderer()`.

==== Logging System

The logging system supports multiple modules and levels:

[source,cpp]
----
fife::Log::log(fife::LOG_LEVEL_DEBUG, "MyModule", "Debug message");
----

Levels: `LOG_DEBUG`, `LOG_INFO`, `LOG_WARNING`, `LOG_ERROR`

Modules can be filtered at runtime via `Log::setModuleLogLevel()`.

===== C++ Logging Macros

When writing C++ code, always use the provided log macros defined in
`engine/core/modules.h`:

[source,cpp]
----
FL_DBG(module, msg)   // Debug level
FL_LOG(module, msg)   // Informational level
FL_WARN(module, msg)  // Warning level
FL_ERR(module, msg)   // Error level
FL_PANIC(module, msg) // Panic level (causes abort)
----

Modules are defined in `<FIFE>/engine/core/modules.h`. Always use the correct
log module when writing to the log.

----

    <Screen width="1024" height="768" />
    <Render backend="OpenGL" />

----

Each XML element maps to a setter in `EngineSettings`:
* `<Screen width="1024">` â†’ `settings.setScreenWidth(1024)`
* `<Render backend="OpenGL">` â†’ `settings.setRenderBackend("OpenGL")`

==== VFS Implementation

The Virtual File System (VFS) abstracts file access:

1. Sources are registered: `vfs.addSource(new DirectorySource("assets/"))`
2. When loading a file, VFS searches sources in registration order
3. Archives (ZIP, etc.) are mounted as sources and accessed transparently
4. The `ImageLoader`, `MapLoader`, and other loaders use VFS for all I/O

==== Cell Cache and Pathfinder

The pathfinder uses a cell cache for efficient lookups:

1. Each layer has a `CellCache` storing walkability, cost, and blocking info
2. When a route is requested, the pathfinder queries the `CellCache`
3. A* search runs on the cell graph (edges defined by `CellGrid` geometry)
4. Results are cached in the `Route` object until invalidated
